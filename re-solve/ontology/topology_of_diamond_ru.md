## СЛОЙ 1 — Классификатор граней

Каждая грань — это **вопрос-атака**. Постулаты записаны как утверждения;
при огранке модель инвертирует их в вопрос и ставит вердикт.

**Переменные:** `c` = Контекст, `g` = Цель, `t` = Инструмент, `v` = Верификация.
`[!]` = топологический модификатор — поведение переменной отличается, см. пометку.

---

### [CORE] — Что это?

```
c → Задача использует концепт в том же объёме, в каком он определён
g → Цель требует именно этого концепта, а не смежного с похожим именем
t → Метод оперирует сущностями именно этого типа
v → Это каноническая формулировка — не реконструированная, не блендированная
```

---

### [DOM] — Где это живёт?

```
c → Домен задачи внутри пространства применимости Бриллианта
g → Цель требует ответа именно в этой системе координат
t → Метод в том же домене — не импортирует чужих допущений
v → Верификация инструментами, валидными в этом домене
```

---

### [MECH] — Как это работает?

```
c → Все входные переменные для механизма присутствуют в контексте
g → Выход механизма совпадает с требуемым целью — по типу и направлению
t → Метод использует механизм корректно: без пропусков шагов, без инверсии
v → Каждый шаг воспроизводим и верифицируем независимо
```

---

### [AUTH] — Почему мы верим?

```
c → Эпистемологический статус источника совместим с парадигмой задачи
g → Уровень доказательности Бриллианта достаточен для цели
t → Метод принимает этот тип основания как валидный вход
v → Утверждение прослеживается до первоисточника без разрывов

[!] Аксиома:  v = «нельзя верифицировать изнутри системы — только явно постулировать»
[!] Гипотеза: v = «отличима от конкурентов по предсказаниям»
```

---

### [CONTR] — Чем это НЕ является?

```
c → Контекст не содержит двойника, маскирующегося под Бриллиант
g → Цель не может быть одинаково закрыта двойником
t → Метод обладает разрешением для различения Бриллианта и двойника
v → Двойник не даёт тот же ответ — рабочий инструмент отбраковки
```

*Примечание:* Именно этот постулат был введён в протокол как грань **(f)**
«ближайший концептуальный двойник».

---

### [LIM] — Когда это ломается?

```
c → Параметры задачи внутри рабочей зоны — не на границе и не за ней
g → Цель не требует ответа именно в точке слома
t → Метод стабилен вблизи граничных условий Бриллианта
v → Граница слома = первичная верификационная ручка; стресс-тест здесь

[!] Эвристика: v = «зоны отказа — главный верификационный тест»
```

*Примечание:* Постулат `c` этой грани введён в протокол как обязательный
позиционный вердикт для граней **(b)** и **(e)**.

---

### [IMPL] — Что из этого следует?

```
c → Следствия не противоречат известным фактам контекста
g → Цель требует следствий Бриллианта, а не самого Бриллианта (или наоборот)
t → Метод умеет распространять следствия данного типа Бриллианта
v → Следствия верифицируются по независимым данным — не по тому же Бриллианту
```

---

### [COST] — Чем мы за это платим?

```
c → Стоимость применения не создаёт запрещённых побочных эффектов
g → Цена применения не обесценивает достижение цели
t → Затраты на метод оправданы качеством результата
v → —
```

---

### [CTRF] — Что рухнет без этого?

```
c → Без Бриллианта часть контекста становится необъяснимой
g → Без Бриллианта путь к цели разрывается — он несущий, не декоративный
t → Метод требует именно этого Бриллианта как входа
v → Отрицание Бриллианта приводит к противоречию с зафиксированными фактами
```

---

### [VER] — Как это быстро проверить?

```
c → Контекст содержит независимую точку данных для чек-суммы
g → Формат инварианта совместим с форматом ожидаемого ответа
t → Метод умеет генерировать этот тип верификационного инварианта
v → Верификатор независим от синтеза — не является выходом того же процесса
```

## СЛОЙ 2 — Топологии: формулы активных связей

**Обозначения:**
- `★` = все четыре переменные активны (c, g, t, v)
- `(x,y)` = только перечисленные переменные активны
- `!` = топологический модификатор (см. пометку [!] в Слое 1)
- Пустая ячейка / отсутствующая переменная = нерелевантно для данной топологии

---

```
[1  ОПРЕДЕЛЕНИЕ / КОНСТРУКТ]
    CORE★ · CONTR★ · LIM★ · AUTH(c,v)

[2  АКСИОМА / ПОСТУЛАТ]
    DOM★ · AUTH(c,t,v!) · CONTR(c,g,t) · LIM(c,v) · CTRF(v)

[3  ТЕОРЕМА / ЛОГИЧЕСКИЙ ЗАКОН]
    MECH★ · LIM★ · AUTH(c,v) · CONTR(c,v) · DOM(c,g)

[4  ЕДИНИЧНЫЙ ФАКТ / ЭМПИРИЧЕСКАЯ КОНСТАНТА]
    CORE★ · AUTH(c,t,v) · LIM(c,v) · CONTR(c,g,v)

[5  МЕХАНИЗМ / СТРУКТУРНАЯ РЕАКЦИЯ]
    MECH★ · CONTR★ · LIM(c,t,v) · DOM(c,t) · IMPL(c,v)

[6  РАБОЧАЯ ГИПОТЕЗА / МОДЕЛЬ ВЕРОЯТНОГО]
    AUTH★ · LIM(c,v!) · CONTR(c,g,v) · IMPL(c,v) · COST(g,t)

[7  СТАТИСТИЧЕСКИЙ ТРЕНД / КОРРЕЛЯЦИЯ]
    CONTR★ · AUTH(c,t,v) · DOM(c,g) · LIM(c,v)

[8  СИНДРОМ / ДИАГНОСТИЧЕСКИЙ ПАТТЕРН]
    CORE★ · CONTR(c,g,v) · MECH(t,v) · LIM(c,v)

[9  КАУЗАЛЬНЫЙ ГРАФ / ИСТОРИЧЕСКИЙ НАРРАТИВ]
    MECH(c,t,v) · AUTH(c,v) · LIM(c,v) · CONTR(c,v)
    ⚠ не один Бриллиант — каждое звено огранять отдельно

[10 ДЕТЕРМИНИРОВАННЫЙ АЛГОРИТМ]
    MECH★ · LIM(c,v) · DOM(c,t) · AUTH(v)

[11 ЭВРИСТИКА / ОПТИМАЛЬНАЯ СТРАТЕГИЯ]
    LIM★(!) · COST(g,t) · CONTR(c,g,v) · DOM(c) · AUTH(c)

[12 НОРМА / ПРЕЦЕДЕНТ]
    DOM★ · AUTH(c,t) · LIM(c,v) · CONTR(c,v)

[13 ИНТЕРПРЕТАЦИЯ / СИМВОЛИЧЕСКИЙ КОНСТРУКТ]
    AUTH(c,g,t) · CONTR(c,v) · LIM(c,v) · DOM(c) · COST(g)

[14 ДИНАМИЧЕСКОЕ ВЗАИМОДЕЙСТВИЕ / РАВНОВЕСИЕ]
    MECH★ · CONTR★ · LIM(c,v) · IMPL(c,v)

[15 АНАЛОГИЯ / СТРУКТУРНЫЙ ИЗОМОРФИЗМ]
    DOM★ · LIM(c,g,v) · CONTR(c,v) · MECH(c,t)
```

## Как использовать при огранке (flow)

```
Кристалл найден
  → определить топологию (строка из Слоя 2)
  → для каждой активной связи найти постулат в Слое 1
  → Domain view: что грань говорит в домене кристалла
  → Task view: инвертировать постулат в вопрос-атаку → вердикт match / mismatch / fragility
```

**Пример (Топология 3 — Теорема, грань [LIM·c]):**
```
Domain view: теорема Бернулли применима при несжимаемом потоке (Re < 2300).
Task view:   задача оперирует сжимаемым газом при M > 0.3 → критерий несжимаемости нарушен.
Вердикт:     MISMATCH → fragility(b),(e)
Действие:    требуется патч [A] или замена топологии на [5 МЕХАНИЗМ].
```
